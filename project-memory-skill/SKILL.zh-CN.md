# 项目记忆

使用这个 Skill 来维护单个项目的持久记忆。目标不是保留全部历史，而是保留一份简短、可靠、便于恢复上下文的项目记忆，让 Agent 和用户在未来都能快速重新进入状态。

## 何时使用

当出现下面这些场景时使用本 Skill：

- 用户希望先把一个项目的背景、目标和当前状态整理清楚，方便后面继续协作
- 用户希望把刚刚的聊天、群聊或讨论内容沉淀成后续还用得上的项目信息
- 用户希望在隔了一段时间后，快速找回一个项目之前的关键上下文
- 用户希望把新的决策、约束、待办、文件位置或其他重要信息补充到项目资料中
- 用户希望 Agent 根据当前对话持续整理项目相关信息，而不是每次都重新理解一遍
- 用户希望为下一次继续这个项目留下清晰的恢复入口、接手说明或提醒重点
- Agent 发现项目背景、目标、确认方或文档存放位置不清楚，需要先向人类问清楚再继续整理

本 Skill 一次只服务一个项目。除非用户明确要求，否则不要把它扩展成多项目记忆系统。

## 核心规则

- 只记录未来大概率仍然重要的信息。
- 将项目记忆视为“恢复上下文的工件”，而不是会议纪要或活动流水账。
- 当无法定位或读取记忆文档时，优先向用户确认其存放位置和实际路径或链接；明确后，将其记录在“记忆文档”栏目中。
- 优先更新已有内容，而不是不断追加重复笔记。
- 保持记忆简短、易扫读、稳定。
- 将项目背景视为高价值且通常较稳定的锚点信息。
- 记录目标时使用 SMART 思维，即使最终不拆成五个子字段。
- 只要人物归因重要，就记录谁是 `负责人`，谁是 `确认方`。
- 如果关键细节模糊、冲突或缺失，在将其写成事实前必须立即向人类发问。

## 什么值得记录

通常值得记录：

- 稳定的项目身份信息
- 项目为什么存在
- 当前目标以及什么算成功
- 当前阶段和最新的关键状态
- 会影响后续工作的决策
- 会限制后续选择的约束
- 重要文件、文档和命令
- 会影响推进的高价值待办
- 帮助未来快速恢复的说明

通常不值得记录：

- 原始且发散的脑暴内容
- 重复性的状态闲聊
- 未被确认的临时猜测
- 之后可以很低成本从代码或文件中重新推导的信息
- 本次会话里的每一个小动作

## 什么时候必须立即发问

如果出现下列情况，先暂停并向人类发问：

- 无法定位或读取记忆文档，或不清楚其存放方式与路径
- 项目背景存在多个可能解释
- 目标过于模糊，无法按 SMART 方式记录
- 缺少成功标准
- 重要决策没有清晰原因，或没有明确确认方
- 某个约束看起来可能存在，但尚未被确认
- 归属关系重要，但没有明确的 `负责人`

只问最少且必要的问题。不要把澄清变成长访谈。先消除歧义，再更新记忆。

## 记忆格式

除非用户明确要求其他格式，否则使用下面这套结构：

```md
# 项目记忆

## 记忆文档
- 存储类型:
- 路径或链接:

## 项目标识
- 名称:
- 负责人 / 相关方:
- 一句话描述:

## 项目背景
- 为什么会有这个项目:

## 目标

## 当前状态
- 当前阶段:
- 已完成内容:
- 下一步预期:

## 关键决策
- 事项:
- 为什么做这个决策:
- 负责人:
- 确认方:

## 约束条件

## 重要参考资料
- 文件名称:
- 文件路径/链接:

## 待办事项
- 事项:
- 负责人:
- 需要在何时前完成:

## 恢复说明
- 恢复时先看这里:
- 需要重新加载的关键信息:
- 容易遗忘的细节:
```

## 各栏目指导

### 记忆文档

这是记忆文档自身的存放信息，不是项目内容本身。开始维护项目记忆前，当无法定位或读取记忆文档时，优先向用户确认：

- 文档存在哪里
- 是云文档还是本地文件
- 该文档的实际链接或路径是什么

如果这些信息不明确，先问再写。不要默认猜测存放位置。

### 项目标识

保持简短且稳定。它应该帮助读者立刻识别这个项目是什么。

### 项目背景

记录项目背后的稳定上下文。聚焦这个项目为什么存在、有什么价值、以及未来会长期影响理解的背景事实。不要把短期更新写在这里。

### 目标

使用 SMART 思维记录目标：

- 尽量具体
- 尽量可衡量
- 与项目现实相匹配
- 与背景保持相关
- 在已知时写出时间预期

如果其中一项或多项缺失，且会影响后续记忆质量，就向人类发问。

### 当前状态

只记录最新且有意义的状态，不要把它写成连续日志。

### 关键决策

每条记录都应说明：

- 决定了什么或确认了什么
- 为什么这样决定
- 和谁确认过

如果事项重要，但 `确认方` 不明确，在把它写入持久记忆前先发问。

### 约束条件

只记录会影响决策、范围、交付或质量的约束。如果某个约束还只是猜测，不要直接写入，除非明确标记为未决问题。

### 重要参考资料

只记录文件路径、链接和文档名称。

### 待办事项

只保留有意义的待办。不要让这个部分被琐碎、短期、低价值任务淹没。

### 恢复说明

这一部分要帮助未来会话快速重启。内容应短、直接、可操作。

## 更新流程

维护项目记忆时，按下面顺序执行：

1. 先查看记忆文档存放位置，并拿到实际路径或链接。
2. 先阅读当前记忆文件。
3. 识别这次新增或变化的持久上下文。
4. 如果关键信息模糊，先提出澄清问题。
5. 更新对应栏目，而不是盲目追加。
6. 在替换信息时删除过时或重复内容。
7. 让最终结果保持足够简洁，便于快速扫读。

## 输出行为

当用户要求管理项目记忆时，执行以下一种行为：

- 按标准格式创建新的记忆文档
- 把当前会话、群聊内容或用户补充整理为可写入记忆的更新，并在已有记忆文档存在时合并到对应栏目中
- 先提出有针对性的澄清问题，再落笔

记忆内容本身应尽量写得干净、直接，避免过程导向的话术。
